# 各行业态势感知产品指南：探针-引擎-平台的三层架构实战

> 面向售前工程师的行业态势感知差异化指南——帮你理解态势感知架构的"不变"与"变"：三层架构不变，探针协议、引擎规则、平台侧重点因行业而异。

---

## 一、核心认知：态势感知的"通用骨架"与"行业血肉"

### 1.1 经典三层架构（所有行业通用）

```
 ┌─────────────────────────────────────────────────┐
 │  ③ 统一管控平台（呈现/分析/响应/合规）              │
 │   ─ SIP/SOC 态势感知平台                          │
 │   ─ 多级级联/大屏展示/工单管理                     │
 └──────────────────────┬──────────────────────────┘
                        │ 关联分析/告警上报
 ┌──────────────────────▼──────────────────────────┐
 │  ② 威胁检测引擎（分析/关联/检测）                  │
 │   ─ 行为分析引擎 / UEBA                          │
 │   ─ 威胁情报匹配引擎                              │
 │   ─ 沙箱检测 / AI 异常检测                        │
 │   ─ 行业专用规则引擎（如工控协议检测）               │
 └──────────────────────┬──────────────────────────┘
                        │ 原始数据采集
 ┌──────────────────────▼──────────────────────────┐
 │  ① 探针层（数据采集）                              │
 │   ─ 流量探针（NTA/NDR）：采集网络流量               │
 │   ─ 端点探针（EDR）：采集主机行为                   │
 │   ─ 日志探针（SIEM采集器）：采集设备日志             │
 │   ─ 行业专用探针（工控/IoMT/数据库）                │
 └─────────────────────────────────────────────────┘
```

**关键认知：** 金融、医疗、政务、制造——四个行业的态势感知平台都遵循这套三层架构。差异不在"要不要装"，而在每一层**装什么、听什么协议、配什么规则**。

### 1.2 常见认知纠偏

| 常见误解 | 技术实情 |
|---------|---------|
| "制造业态势感知要用工控探针" | **只说对了一半。** 制造业分为IT办公网和OT生产网。**办公网**（ERP、财务、邮件）用的依然是传统IT探针（NTA/EDR）；只有**OT生产网**（车间、流水线）才需要工控协议探针。一套完整的制造业态势感知，往往是"IT探针+OT探针"混合部署 |
| "金融行业只需要高性能探针" | 高性能只是基础。金融行业的核心需求是**反欺诈规则引擎**和**数据库审计探针**——金融客户不关心网络通不通，关心的是"有没有人在偷数据、洗钱、撞库" |
| "医疗行业探针和通用探针一样" | 医疗IoMT设备（CT/MRI/监护仪）大量使用**专有协议（HL7、DICOM、MLLP）**，通用探针无法解析这些流量，必须用医疗专用探针 |
| "政务平台只要大屏好看" | 政务态势感知平台的核心能力是**多级级联**——省级平台要能直接调取地市级、区县级的数据，实现"通报-整改-反馈"闭环 |

## 二、行业差异化矩阵一览表

| 维度 | 工业制造 | 金融行业 | 医疗行业 | 政府机构 |
|------|---------|---------|---------|---------|
| **核心保护资产** | PLC/DCS/生产线 | 核心交易系统/资金数据库 | 医疗仪器(IoMT)/病历数据 | 政务外网/门户网站/公民隐私 |

| **应急响应核心** | 恢复生产（分钟级） | 止血止损（秒级） | 隔离设备防扩散（分钟级） | 溯源反查（小时级） |
| **预算层级** | 50-200万 | 200-1000万+ | 80-300万 | 100-500万 |

## 三、行业深度分解

### 3.1 工业制造 —— "生产不停，安全不退"

#### 行业特征

制造业数字化程度两极分化严重：头部工厂（汽车/电子）已实现高度自动化，小厂仍在使用老旧工控设备。攻击者一旦通过IT网跳板进入OT网，可导致产线停摆甚至设备物理损坏。

#### 态势感知架构详解

```
┌─ OT生产网 ──────────────────────────────────────────┐
│  平台层：工控态势感知平台                             │
│  ├─ 生产可视化大屏（产线状态+安全事件叠加）            │
│  ├─ OT资产测绘（PLC/DCS/SCADA/HMI自动识别）          │
│  └─ IT/OT统一管理（可选）                            │
│                                                      │
│  引擎层：工控威胁检测引擎                             │
│  ├─ 工控协议深度解析（Modbus/TCP、S7、Profinet、EtherCAT）│
│  ├─ 工艺基线异常检测（学习正常生产流量模式）            │
│  ├─ PLC指令篡改检测（禁止非授权写寄存器）              │
│  └─ 设备违规外联检测（如PLC私自拨号上网）              │
│                                                      │
│  探针层：                                              │
│  ├─ 工控网络探针（旁路镜像采集OT流量）                 │
│  └─ OT端点探针（老旧设备用Agentless扫描）             │
└──────────────────────────────────────────────────────┘

┌─ IT办公网 ──────────────────────────────────────────┐
│  使用通用安全产品（EDR/NTA/传统防火墙）               │
│  保护ERP、OA、邮件系统                               │
└──────────────────────────────────────────────────────┘
```

#### 探针协议差异

| 工业协议 | 用途 | 检测关注点 |
|---------|------|-----------|
| **Modbus/TCP** | PLC通信 | 功能码异常（如停止指令）、寄存器越界读写 |
| **S7 (Siemens)** | 西门子PLC | 上传/下载程序锁、DB块篡改 |
| **Profinet** | 实时工业以太网 | 周期报文异常、设备身份仿冒 |
| **EtherCAT** | 高速运动控制 | 从站断连、过程数据篡改 |
| **OPC UA/DA** | 上位机通信 | 订阅关系异常、数据采集中断 |

#### 售前话术

> "老师，工厂安全跟写字楼不一样——写字楼被黑了是丢数据，工厂被黑了是产线停转、一天损失几百万。我们的方案分两层：**办公网**用传统安全产品保ERP、保财务数据；**生产网**用工控探针+工控引擎，专门监听PLC和DCS之间的指令。一旦发现有人在改流水线的温度参数或者下发非法停止指令，系统直接告警+自动阻断，保产线不停。而且引擎会自动学习正常的生产流量模式，知道什么是'正常干活'、什么是'被黑客操控'，误报率极低。"

#### 异议应对

| 客户异议 | 应对话术 |
|---------|---------|
| "工控网跟互联网是物理隔离的，不用安全" | "物理隔离靠的是人自觉。2021年Colonial Pipeline就是通过VPN进入OT网的。物理隔离挡不住钓鱼邮件+跳板攻击。" |
| "我们PLC太老了，装不了探针" | "我们的工控探针是旁路部署，像网络摄像头一样光看不说话，不装软件不影响老旧PLC运行。" |

### 3.2 金融行业 —— "秒级止血，分秒必争"

#### 行业特征

金融行业（银行/证券/保险）是黑客攻击的头号目标。交易系统7x24小时运行，每秒处理上万笔交易。安全的核心目标是：**防数据泄露、防资金被盗、防业务中断**。监管严格程度全行业第一。

#### 态势感知架构详解

```
┌─ 统一管控平台 ──────────────────────────────────────┐
│  金融态势感知平台（SOC/SIEM+SOAR）                    │
│  ├─ 极速响应联动：检测到威胁→自动封禁IP→阻断交易      │
│  ├─ 合规报告：一键生成等保/银保监会/人行合规报表        │
│  ├─ 反欺诈引擎联动：将安全告警与交易流水交叉关联        │
│  └─ 总分架构：总行SOC监管所有分行/支行态势             │
│                                                      │
│  威胁检测引擎层                                        │
│  ├─ 高性能关联分析引擎（百万级EPS — 事件/秒）          │
│  ├─ 反洗钱/反欺诈规则引擎（交易行为分析）              │
│  ├─ 数据库行为分析（谁/何时/从哪/查了哪个表）          │
│  ├─ API安全检测（识别API滥用、异常调用频率）            │
│  └─ 沙箱检测（防范钓鱼邮件中的未知恶意软件）            │
│                                                      │
│  探针层                                              │
│  ├─ 骨干网万兆/十万兆探针（全端口流量采集、零丢包）      │
│  ├─ 数据库审计探针（SQL操作100%记录）                  │
│  ├─ WAF+API网关探针（Web/移动端流量过滤）              │
│  └─ 邮件网关探针（钓鱼邮件检测）                      │
└──────────────────────────────────────────────────────┘
```

#### 探针协议差异

| 协议/接口 | 用途 | 检测关注点 |
|----------|------|-----------|
| **HTTP/HTTPS** | 网上银行/移动端 | SQL注入、XSS、API滥用、撞库攻击 |
| **ISO 8583** | 金融交易报文 | 交易篡改、重放攻击、异常金额 |
| **JDBC/SQL*Net** | 数据库访问 | 脱库、慢查询、权限越权 |
| **FIX协议** | 证券交易 | 订单异常、市场操纵 |
| **LDAP/RADIUS** | 认证系统 | 暴力破解、凭证窃取 |

#### 售前话术

> "领导，金融行业最大的特点是——**数据就是钱**。丢了客户资料，银保监会罚款动辄上亿。我们为金融客户设计的是**'高性能探针+SOAR极速响应'组合**：骨干网上用万兆探针保证零丢包、零延时；检测引擎不仅看攻击，还关联交易流水，发现撞库、洗钱行为直接联动阻断。而且我们的平台能**一键出合规报告**——等保、银保监会、人行检查，点一下就有。"

#### 异议应对

| 客户异议 | 应对话术 |
|---------|---------|
| "我们已有SIEM了" | "现有的SIEM管合规可以，但实战响应不够。我们在SIEM之上加SOAR+XDR的能力，检测到威胁自动联动防火墙+WAF阻断，不用人等，真正实现秒级止血。" |
| "高性能探针太贵" | "金融系统停摆1分钟的损失超过探针价格。而且我们探针支持智能采样+关键交易全量捕获，兼顾性能和成本。" |

### 3.3 医疗行业 —— "设备不能停，隐私不能丢"

#### 行业特征

医疗行业最大的安全痛点是**医疗设备（IoMT）的先天脆弱性**——CT机、MRI、监护仪普遍使用Win 7/XP甚至更老系统，不能打补丁、不能装杀毒软件。同时患者病历（EMR/PACS）在黑市价值极高。勒索病毒是头号威胁。

#### 态势感知架构详解

```
┌─ 统一管控平台 ──────────────────────────────────────┐
│  医疗态势感知平台（极简运维型）                        │
│  ├─ 资产可视化大屏：全院IoMT设备地图，一眼看全          │
│  ├─ 设备脆弱性看板：哪台设备系统多老、有没有异常流量    │
│  ├─ 勒索病毒专项告警：横向传播行为自动识别              │
│  ├─ 外包运维审计：厂商远程操作全程录像+权限管控         │
│  └─ 等保合规报告：一键生成医疗等保扩展合规报表          │
│                                                      │
│  威胁检测引擎层                                        │
│  ├─ IoMT设备行为基线学习（设备正常通信模式）            │
│  ├─ 勒索病毒横向传播检测（SMB暴力破解/WMI远程执行）     │
│  ├─ DICOM协议异常检测（非法影像访问/导出）             │
│  ├─ PACS异常访问检测（大批量拉取影像=数据泄露）         │
│  └─ 弱口令/默认口令扫描（设备出厂密码不改）            │
│                                                      │
│  探针层                                              │
│  ├─ 医疗IoT流量探针（旁路采集，解析HL7/DICOM/MLLP）    │
│  ├─ 内网流量探针（监控医疗设备间的异常通信）            │
│  ├─ 堡垒机探针（厂商运维行为捕获）                    │
│  └─ 邮件安全网关（防钓鱼邮件——勒索病毒的常见入口）      │
└──────────────────────────────────────────────────────┘
```

#### 探针协议差异

| 医疗协议 | 用途 | 检测关注点 |
|---------|------|-----------|
| **HL7 v2/v3** | HIS系统间患者数据交换 | 患者信息批量查询（数据泄露）、非法修改 |
| **DICOM** | 影像设备（CT/MRI）通信 | 影像批量导出（数据泄露）、非授权设备读取 |
| **MLLP** | HL7的低层传输协议 | 通信中断、消息篡改 |
| **POCT** | 床旁检测设备 | 设备认证绕过、数据伪造 |
| **IEEE 11073** | 个人健康设备 | 设备身份仿冒、隐私数据传输 |

#### 售前话术

> "老师，医疗行业态势感知跟别的行业完全不同——核心不是保数据，是**保设备、保运行**。CT机停了，急诊病人没法做检查。我们的方案用一种**'不打扰'的方式**：探针旁路部署，不往设备上装软件，不扫描老旧系统。引擎会先学习每台设备的'正常行为'——比如这台CT机平时只跟PACS服务器通信，突然它开始大规模扫内网，那基本是中勒索病毒了。平台大屏上，全院所有IoMT设备一目了然，哪台设备快不行了一看就知道。"

#### 异议应对

| 客户异议 | 应对话术 |
|---------|---------|
| "我们设备太杂，品牌型号太多" | "这正是我们探针的强项——自动指纹识别，支持3000+医疗设备型号识别，不需要手工录入。" |
| "医院预算紧张" | "那就先做**资产梳理+横向移动检测**两个核心模块，投入最小、效果最明显。防住勒索病毒一次，省下的钱抵得上10年的安全预算。" |

### 3.4 政府机构 —— "防篡改、防通报、防重保丢分"

#### 行业特征

政府机构的安全驱动力不是商业利益，而是**政治影响**和**监管排名**。核心关切：网站不能被篡改（防止出现反动言论）、敏感数据不能泄露（人口/地理/社保数据）、重保期间不能丢分（公安部/网信办排名通报）。

#### 态势感知架构详解

```
┌─ 统一管控平台 ──────────────────────────────────────┐
│  政务态势感知平台（多级级联型）                        │
│  ├─ 省-市-县三级级联：上级可调取下级安全数据            │
│  ├─ 通报预警模块：自动生成安全通报，下发整改工单        │
│  ├─ 重保指挥调度大屏：HVV/两会期间专供                  │
│  ├─ 排名考核系统：地市安全状况自动打分排名              │
│  └─ 数据上报接口：对接网信办/公安厅监管平台             │
│                                                      │
│  威胁检测引擎层                                        │
│  ├─ 网页篡改检测（内核级文件驱动保护）                  │
│  ├─ 暗链/后门检测（Webshell + 隐藏页面扫描）           │
│  ├─ APT潜伏检测（长期低慢行为分析）                    │
│  ├─ 敏感数据泄露检测（身份证/手机号/住址出境监控）      │
│  └─ DDoS攻击检测（政务网站高并发清洗）                 │
│                                                      │
│  探针层                                              │
│  ├─ 网络全流量回溯探针（数据包完整保存7-30天供溯源）    │
│  ├─ 网页完整性探针（分钟级检测网站文件是否被改）        │
│  ├─ 蜜罐探针（虚假服务器+假数据库诱捕攻击者）           │
│  └─ 主机进程监控探针（检测未知后门/挖矿程序）           │
└──────────────────────────────────────────────────────┘
```

#### 探针关键特性

| 政务场景 | 技术方案 | 为什么重要 |
|---------|---------|-----------|
| **网页防篡改** | 内核级文件驱动保护 | 网页被篡改后100ms内恢复，对访问者完全无感，防止产生政治影响 |
| **全流量回溯** | 数据包完整存储+索引 | APT攻击往往数周后才被发现，需要回溯当时的所有流量进行完整取证 |
| **蜜罐诱捕** | 高交互蜜罐+攻击画像 | 捕获黑客的社交账号/IP/工具链，支撑公安反制，同时产生"破案"成果向上汇报 |
| **多级级联** | 省-市-县三级数据汇聚 | 省公司能看到全省安全态势，直接下发通报到具体地市，地市整改后反馈闭环 |

> "领导，政务安全跟企业思路不一样——企业怕丢钱，政务怕**丢分、丢面子**。我们的政务态势感知平台围绕三个痛点来设计：第一，**网页防篡改**——门户网站被黑了，内核级保护自动恢复，访问者完全觉察不到，不给上级通报的机会。第二，**全流量回溯**——APT攻击的发现窗口期很长，我们的探针能保存30天完整流量包，出了事随时回溯取证。第三，**多级级联监管**——省公司一张大屏看全省安全态势，自动生成通报给地市，地市整改后系统自动反馈闭环。网信办来检查，报告一键导出。"

## 七、总结

> **态势感知的"不变"：** 所有行业都遵循"探针采集 → 引擎分析 → 平台呈现"的三层架构。不存在某个行业"不需要态势感知"的情况——区别只在于规模和复杂度。
>
> **态势感知的"变"：**
> - **探针层**跟着**协议**走——工业听Modbus，金融听ISO 8583，医疗听HL7/DICOM，政务听HTTP全流量
> - **引擎层**跟着**威胁**走——工业防指令篡改，金融防交易欺诈，医疗防勒索横向，政务防网页篡改和APT潜伏
> - **平台层**跟着**管理**走——工业重生产连续性，金融重秒级响应，医疗重资产可视化，政务重多级通报
>
> **一句话：选对探针是基础，配对引擎是核心，建对平台是保障。知道客户行业，就知道三层怎么配。**
