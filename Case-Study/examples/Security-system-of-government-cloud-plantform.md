# 售前案例学习笔记：某市政务云平台安全体系升级

> *文末附有我的复盘思考。*

|------|------|

## 二、客户背景

- **现状描述**：政务云已运行3年，承载各委办局业务系统。近期省里组织网络安全检查，发现政务云安全能力不均衡——部分委办局安全意识弱，系统长期未更新补丁，存在多个高危漏洞。大数据管理局作为政务云运营方，需要对入驻单位的安全状况进行统一监管。
- **已有安全投入**：政务云基础安全资源池（防火墙、WAF、堡垒机），但安全能力按资源池统一部署，无法按委办局差异化配置。安全运营由3人团队负责。
- **行业监管要求**：《网络安全法》《关键信息基础设施安全保护条例》《政务信息系统安全管理要求》，等保三级（核心系统）/二级（一般系统）。

---

## 三、我的思考

- **打动客户的点**："安全评分排名通报"——这是政府客户最能理解的推动方式，技术解决不了的问题用管理手段解决
- **简化表达**：统一平台可视 + 排名通报推动 + 利旧增强，三件事讲清楚方案价值
- **可迁移场景**：省级政务云、区县级政务云、教育城域网、医疗联合体
- **竞品攻击点**："统一管理平台各家都能做，你们优势在哪？" 防守："我们的优势是内置等保合规模块和政务场景适配——委办局安全评分模型是专门设计的，不是通用方案改的"
- **学到的技巧**：政府行业要懂行政逻辑——安全方案不仅要技术可行，还要符合政府的管理方式（通报、排名、考核），否则推不动

---

## 四、痛点分析

**一句话痛点：** 80多个委办局系统安全水平参差不齐，省里检查出高危漏洞，3人团队管不过来。

| 表面诉求 | 深层痛点 | 翻译为技术需求 |
|----------|---------|--------------|
| "省里网络安全检查发现多个高危漏洞" | 委办局安全意识薄弱，系统长期不更新，漏洞堆积 | 需要自动化漏洞扫描 + 补丁管理 + 漏洞整改跟踪闭环 |
| "80多个委办局，安全水平参差不齐" | 有的委办局很重视，有的完全不管，一刀切方案行不通 | 需要按委办局差异化安全策略 + 安全评分排名通报机制 |
| "大数据局只有3人管安全" | 运营方责任大但人手少，无法逐一管理80个单位 | 需要统一安全管理平台 + 自动化巡检 + 委办局自助服务 |
| "政务云已有安全资源池" | 现有投资不能浪费，需要利旧 + 增强 | 需要在现有资源池基础上做能力增强，不是推倒重来 |

---

## 五、解决方案

| 产品/服务 | 解决什么问题 | 为什么选这个 |
|-----------|------------|-------------|
| 统一安全管理平台（利旧+增强） | 80个委办局安全状况统一可视、统一管理 | 解决"3人管80个单位"的能力瓶颈 |
| 自动化漏洞扫描 + 整改跟踪 | 定期扫描、自动派单、整改验收闭环 | 解决"漏洞堆积无人管"的问题 |
| 委办局安全评分排名系统 | 按安全评分排名通报，督促整改 | 用行政手段推动安全工作，适合政府场景 |
| WAF策略优化（现有设备调优） | 针对政务系统特点的WAF规则定制 | 利旧投资，快速见效 |
| 安全运营MSS服务 | 7×24监控 + 专家分析，弥补3人团队不足 | 释放人力聚焦管理平台运营 |

**方案亮点：**

- **利旧优先**——在现有安全资源池基础上增强，不推倒重来，节约财政预算
- **管理平台为核心**——不是堆产品，而是建能力，用平台管80个委办局
- **行政手段+技术手段结合**——安全评分排名通报，用政府的管理逻辑做安全
- **等保合规内置**——平台内置等保二/三级检查清单，一键生成合规报告

## 七、可复用的面试话术

```markdown
"我之前接触过政府行业的一个项目，客户是某地级市大数据管理局，

他们遇到的核心问题是：
安全能力不是不够，而是不均衡——有的单位做得好，有的完全不做；

省里检查压力大，需要快速整改。

我们当时给的方案核心是两件事：
第一，建统一安全管理平台，80个委办局安全状况统一可视；
第二，用行政手段推动安全——安全评分排名通报，排名靠后的主动整改。
技术上配合自动化漏洞扫描 + MSS托管运营，弥补3人团队的能力不足。

核心思路是利旧优先——在现有安全资源池基础上增强，

上线结果：下次省检查0高危漏洞顺利通过，
漏洞平均修复时间缩短到7天，委办局安全意识整体提升。

现在安全状况天天看得到，排名靠后的主动来找我们要整改方案。'

这个案例让我对政府行业的安全需求有了深刻理解，总结下来就是：
政府行业安全的核心逻辑是'管理驱动'——技术手段要配合行政手段，
用排名通报、考核问责来推动安全工作落地。
"
```
