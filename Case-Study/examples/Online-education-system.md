## 二、客户背景

*客户在什么状态下找到厂商？原来有什么？*

- **现状描述**：收到安全通报，一个API接口泄露了30多万条用户数据（姓名、手机号、学习记录）。自查发现是开发为了方便测试，把debug接口留在生产环境且没加鉴权。老板非常生气，要求一个月内整改到位。
- **已有安全投入**：未明确提及，推测基础防护有但API安全层面存在盲区。
- **行业监管要求**：教育行业涉及大量学生个人信息，需满足《个人信息保护法》《数据安全法》及等保2.0合规要求。

## 三、我的思考

- **这个案例最打动客户的点是什么？**
  "1-3天内出事件响应报告"——客户现在"老板非常生气"，最需要的不是长期方案，而是能立刻给老板交代的东西。先灭火再谈建设，节奏对了客户自然信任你。

- **如果我来讲这个案例，我会怎么简化/突出重点？**
  核心就三句话：
  1. 问题根源是流程缺失，不是产品不够——测试接口没下线、没鉴权
  2. 方案分层递进——先止血（1-3天）→ 再排查（3-7天）→ 后建流程
  3. 效果一句话——"以前出了事才查，现在上线前就能拦住"

- **这个案例能迁移到哪些其他行业场景？**
  任何"业务跑得快、安全跟不上"的行业都适用：互联网创业公司、电商平台、SaaS企业、金融科技。背后的共性问题是"研发效率与安全合规的平衡"。

- **如果竞品来抢单，最可能的攻击点是什么？怎么防守？**
  竞品攻击点："你们不是也要推DLP和MSS吗？跟其他厂商有什么区别？"
  防守方式："我们前两步（止血+排查）不推任何产品，先帮您把问题搞清楚。DLP和MSS是长期选项，您觉得有需要了再聊。"——这个立场是竞品最难复制的。

- **我学到了什么可以复用的售前技巧？**
  - **按场景调整询问顺序**——危机场景先问1-2个关键信息再给框架；常规场景先问完再出方案；探索场景先给价值再跟进
  - **方案要有时间线**——"1-3天""3-7天"比"尽快""及时"有说服力十倍
  - **产出物要具体**——事件响应报告、风险清单、修复优先级，让客户看得见摸得着
  - **先解决根本原因**——客户问题是鉴权缺失，首推DLP就偏了方向

## 四、痛点分析

*用一句话说清楚客户的核心问题。*

**一句话痛点：** 开发留了个没鉴权的debug接口在生产环境，数据被爬了30多万条，老板追责、人手不够、时间还紧。

| 表面诉求 | 深层痛点 | 翻译为技术需求 |
|----------|---------|--------------|
| "一个API接口泄露了30多万条用户数据" | 不是单纯的技术漏洞，而是研发流程缺失——测试接口上线没走安全检查流程 | 紧急封堵泄露接口 + 排查其他残留测试接口 + 建立上线前安全检查流程 |
| "安全团队就2个人，根本搞不过来" | 安全能力与业务规模严重不匹配，日常运维已饱和，无暇处理专项整改 | 需要轻量级、自动化的排查工具，以及可选的MSS托管服务分担人力压力 |
| "老板要求一个月内整改到位，下季度还要上线新课程平台" | 两线作战：既要处理安全事故，又要保障新项目按时上线 | 需要分层方案：先止血（1-3天）→ 再排查（3-7天）→ 长期建设（可选） |
| "开发为了方便测试把debug接口留到生产环境" | 研发安全意识薄弱，缺乏API资产清单和下线机制 | 需要API资产盘点 + 接口鉴权规范 + 开发安全培训 |

## 五、解决方案

*厂商推荐了什么？*

| 产品/服务 | 解决什么问题 | 为什么选这个 |
|-----------|------------|-------------|
| 紧急止血服务（远程，1-3天） | 立即封堵泄露接口，排查其他暴露接口 | 不采购任何产品，立刻启动，先灭火 |
| API资产盘点 + 漏洞扫描（3-7天） | 梳理所有接口、测试接口、未鉴权接口 | 自动化工具辅助，减轻2人团队负担 |
| 《数据泄露事件响应报告》 | 给老板汇报的材料，交代整改进展 | 满足"一个月内给说法"的管理需求 |
| 《API安全风险清单》 | 问题接口清单 + 风险等级 + 修复优先级 | 指导修复工作，按轻重缓急排序 |
| API安全网关 / WAF策略调优（可选） | 接口级防护，防未授权访问 | 轻量级，解决接口安全核心问题 |
| DLP数据防泄露系统（可选） | 防敏感数据被批量导出 | 治标层面的最后一道防线 |
| MSS安全运营托管（可选） | 7×24安全监控，分担2人团队压力 | 解决"人手少"的根本困境 |

**方案亮点**（什么让这个方案脱颖而出）：

- **先止血再排查**——不急着推产品，先帮客户把火烧灭，建立信任后再谈长期建设
- **分层递进，时间明确**——紧急止血（1-3天）→ 全面排查（3-7天）→ 修复加固（1-2周）→ 长期建设（可选），每步都有明确时间节点
- **量化产出物**——事件响应报告、API风险清单、修复优先级列表，让客户看得见、摸得着
- **不否定已有团队**——不说"你们安全团队不行"，而是说"你们已经做得很好了，只是需要工具和流程支撑"
- **兼顾新项目上线**——方案中考虑了下季度新课程平台的上线时间节点

## 七、可复用的面试话术


```markdown
"我之前接触过教育行业的一个项目，客户是一家有300多台服务器的在线教育公司，
开发团队60多人，但安全只有2个人。他们遇到的核心问题是：
开发为了方便测试，把一个debug接口留在生产环境没加鉴权，
结果被爬了30多万条用户数据，老板非常生气，要求一个月整改到位。

我们当时给的方案分三步走：
第一步，紧急止血，1-3天内远程封堵泄露接口，排查其他暴露接口，
同时出具事件响应报告让客户能跟老板交代；
第二步，全面排查，3-7天完成API资产盘点和漏洞扫描，
产出风险清单指导修复；
第三步，长期建设，根据客户预算选择API安全网关、DLP或MSS托管。

核心思路是先解决接口鉴权这个根本原因，再谈DLP等防流出措施。
我们还帮他们建立了'开发-测试-上线'的安全检查流程。

上线结果：第一天就封堵了泄露接口，一周内梳理出所有测试接口的下线清单。
他们技术负责人的原话是——'以前是出了事才查，现在是上线前就能拦住。'
后来新课程平台上线，整个过程没再出安全事件。

这个案例让我对教育行业的安全需求有了深刻理解，总结下来就是：
教育行业安全的核心矛盾不是'缺不缺产品'，而是'研发流程缺不缺安全检查'——
业务跑得快，安全得跟上节奏。
"
```

## 附录：本项目沉淀的新知识

### 售前询问顺序按场景灵活调整

这是基于本项目学习沉淀的新方法论，核心是**匹配客户心态，而非死板执行流程**。

**三种场景的快速判断法：**

| 客户话术关键词 | 判断场景 | 采用顺序 |
| --- | --- | --- |
| "已经出事了""老板生气""一周内""整改" | **危机场景** | 轻量反问 → 初步方案 → 详细反问 |
| "想优化""最近在评估""准备上线" | **常规场景** | 完整反问 → 完整方案 → 预算 |
| "了解一下""介绍一下""有什么产品" | **探索场景** | 价值展示 → 轻量反问 → 跟进 |

**核心原则：**
- 客户焦虑，先止血再诊断
- 客户理性，先诊断再出方
- 客户观望，先给值再跟进

> 顺序对了，客户自然信任你。
